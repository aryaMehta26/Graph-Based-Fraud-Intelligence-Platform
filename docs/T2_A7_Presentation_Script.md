## SLIDE 2 — The Problem
**Title:** The Problem With Traditional AML

**Left panel (red border — THE PROBLEM):**
Traditional AML systems look at one transaction at a time.
They fail to detect:
- **Fraud rings** — coordinated accounts acting together
- **Funnel accounts** — many small feeds into one hub
- **Layering chains** — money hopping A → B → C to obscure origin
- **Cyclic flows** — money returning to origin through a loop

> 95%+ of alerts generated by rule-based systems are false positives.
> Investigators are drowning in noise.

**Right panel — 4 stat boxes (orange):**
- `31.9M` Transactions
- `1.76M` Accounts
- `35,230` Fraud transactions
- `2,756` Fraud rings

**Speaker note:** The IBM dataset gives us ground truth for all 2,756 rings — rare for AML benchmarks.

## SLIDE 6 — Layer 1: XGBoost Baseline
**Title:** Layer 1 — XGBoost Baseline (Transaction Level)

**Left — WHY XGBoost:**
- Scales to 22.3M training rows efficiently
- Handles 904:1 class imbalance via `scale_pos_weight=904`
- Native SHAP integration for feature explainability
- Faster than Random Forest, more interpretable than GNNs

**Right — Model Configuration:**
```
XGBClassifier
  n_estimators:      500
  max_depth:         6
  scale_pos_weight:  904
  eval_metric:       aucpr
  early_stopping:    50 rounds
  threshold:         0.30 (tuned on val set)
```

**Bottom — 11 Tabular Features:**
`log_amount · amount_bucket · hour · day_of_week · is_weekend`
`is_ACH · is_Wire · is_Cheque · is_CC · is_Bitcoin · is_cross_currency`

**Speaker note:** We chose PR-AUC over accuracy because a model predicting "legitimate" for everything gets 99.89% accuracy but catches zero fraud.

## SLIDE 7 — Layer 2: Graph Centrality
**Title:** Layer 2 — Graph Centrality (Account Level)

**Left — The Intuition:**
```
Account A → Account B ←── Account C
                ↑
           Account D

Account B: in_degree = 3 → Funnel!
```
Normal people send to 2–5 accounts.
A laundering hub sends to hundreds.

**Right — What We Compute (per account, from Neo4j):**
| Feature | Description |
|---------|-------------|
| `src_in_degree` | Accounts sending TO this sender |
| `src_out_degree` | Accounts this sender sends TO |
| `src_total_degree` | In + Out |
| `src_degree_centrality` | total_degree / (N−1) |
| + 4 matching `dst_` features | For the receiver account |

**Stats:**
- 1,758,573 accounts profiled
- 123,342 accounts with degree > 100 (hub tier)
- Extracted via Cypher queries — **no GDS plugin required**

## SLIDE 8 — Layer 3: Leiden Community Detection
**Title:** Layer 3 — Leiden Community Detection (Ring Level)

**Left — Louvain vs Leiden:**
| | Louvain | **Leiden** ✓ |
|-|---------|------------|
| Speed | O(n log n) | O(n log n) |
| Connected communities? | ✗ Not guaranteed | ✓ Guaranteed |
| Why it matters for AML | Can inflate fraud rates of innocent accounts | Accurate guilt-by-association |

**We chose Leiden** because a disconnected community would wrongly flag innocent accounts — critical in AML.

**Right — What It Produces:**
```python
# 4 community features per transaction:
community_id          # which ring
community_size        # how many accounts
community_fraud_rate  # % of transactions that are fraud (sender)
community_fraud_rate  # same for receiver
```

**Stats box:**
- `66,740` communities detected
- `27` communities with fraud rate > 50%
- Runtime: **~4 minutes** on 1.76M accounts
- Implemented in Python (igraph) — no Neo4j GDS needed

**Key insight (highlighted box):**
> *A $500 transfer looks very different when the sender's community has a 42% fraud rate.*
> *That's guilt-by-association — and it works.*

## SLIDE 10 — Why This Approach (Justification)
**Title:** Model Selection Rationale

**Table — full comparison:**
| Dimension | XGBoost | Centrality | Leiden | LLM |
|-----------|---------|-----------|-------|-----|
| Level | Transaction | Account | Community | Case |
| Labels needed? | Yes | No | No | No |
| Explainable? | SHAP values | Centrality scores | Community membership | Natural language |
| What it misses | No graph context | No ring structure | No human narrative | Needs upstream quality |
| Innovation | Baseline benchmark | Neo4j + igraph (no GDS) | Leiden over Louvain | 4-variant prompt evaluation |

**Why not Graph Neural Networks?**
- Require large labeled subgraphs — we only have transaction-level labels
- Training time prohibitive on 31.9M transactions
- XGBoost + graph features achieves competitive performance with full interpretability
